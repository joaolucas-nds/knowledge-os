# Log — 2026-06-05

## Objetivo da sessão
Sessão inaugural do projeto Knowledge OS. Objetivo: benchmark do mercado, definição de escopo, arquitetura técnica completa e projeção da árvore de componentes React do canvas.

## Feito

### Benchmark e análise de mercado
- Avaliadas 14 ferramentas (Milanote, Miro, Draw.io, Lucidchart, Canva, FigJam, Figma, Mural, Obsidian Canvas, Heptabase, Whimsical, Scapple, Notion, Logseq).
- Classificados modelos de preço: Gratuito para Sempre (Draw.io, Obsidian Canvas, Logseq), Pago (Heptabase, Scapple), Falso Grátis (todos os demais).
- Identificado gap: nenhuma ferramenta entrega simultaneamente notas ricas + canvas + tabelas + busca semântica sem tier pago.

### Arquitetura de banco de dados
- Definido modelo "tudo é um Block" como entidade universal.
- Schema SQL completo projetado: `blocks`, `relations`, `canvas_nodes`, `canvas_edges`, `database_schemas`, `search_vectors`.
- Decidido wa-sqlite + OPFS para persistência local no browser.
- Decidido Automerge 2.0 para sincronização CRDT.
- Decidido pgvector para busca semântica no backend.

### Engenharia do canvas infinito
- Definida integração tldraw v3 como motor.
- Projetados NoteShapeUtil (Tiptap via HTMLContainer) e DatabaseShapeUtil.
- Definida estratégia de frustum culling com RBush e LOD (full | thumb | dot).
- Definido sistema de edges com pathfinding A* em camada SVG absoluta.

### Sistema de busca global
- Definida arquitetura em 3 camadas: Orama (< 5ms, client-side) + FTS5 SQLite (offline fallback) + pgvector (semântico, online).
- Definida fusão por RRF (Reciprocal Rank Fusion) para combinar keyword e semântico.
- Escolhido modelo text-embedding-3-small da OpenAI para embeddings.
- Implementação de chunking: 512 tokens com overlap de 50.

### Arquitetura de componentes React do canvas
- Projetados 10 componentes: CanvasApp, CanvasShell, CanvasView, NoteShapeUtil, DatabaseShapeUtil, CanvasEdges, CanvasToolbar, NodeInspector, SearchPalette, Sidebar.
- Criado widget interativo HTML navegável com props, hooks e fluxo de dados por componente.
- Criado diagrama SVG de camadas (Provider Root → tldraw Core → SQLite/CRDT/Search).

### Documentação do projeto
- Gerados todos os arquivos de contexto: CONTEXT, STATUS, DECISIONS (7 entradas), CHANGELOG, IDEAS, ROADMAP (6 fases), GLOSSARY, HISTORICO, LOG-TEMPLATE, logs/2026-06-05.

## Decisões
- DEC-001: tldraw v3 como motor de canvas (vs. Konva, React Flow, Excalidraw, construção própria).
- DEC-002: "Tudo é um Block" como entidade universal (vs. tabelas separadas por tipo).
- DEC-003: wa-sqlite + OPFS para persistência local (vs. IndexedDB, localStorage, sql.js, PouchDB).
- DEC-004: Automerge 2.0 para CRDT (vs. Yjs, Last-Write-Wins, OT).
- DEC-005: Orama 3.x para busca client-side (vs. Minisearch, Fuse.js, Lunr.js).
- DEC-006: Busca híbrida com RRF (vs. só keyword, só semântico, Elasticsearch).
- DEC-007: Monorepo Turborepo (vs. repos separados, Nx).

## Aprendizados / armadilhas
- Armadilha registrada no CONTEXT: não usar localStorage para persistência (5MB, síncrono).
- Armadilha registrada no CONTEXT: não renderizar todos os nós do canvas sem frustum culling.
- Armadilha registrada no CONTEXT: não editar `crdt_state` diretamente no banco.
- Armadilha registrada no CONTEXT: não duplicar conteúdo de Block em `canvas_nodes`.
- Armadilha registrada no CONTEXT: tldraw store ≠ Zustand store (não misturar).
- Armadilha registrada no CONTEXT: embeddings devem ser assíncronos com debounce, nunca por keystroke.
