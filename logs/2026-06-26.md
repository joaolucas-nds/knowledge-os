### Parte 2 — Pivô arquitetural (online-first, single-user)
- Usuário perguntou sobre usar Supabase/Cloudflare R2 para tornar o projeto acessível de máquinas diferentes, com login.
- Pesquisado e esclarecido: Supabase (Postgres+Auth+Storage, free tier 500MB banco/1GB storage, pausa após 7 dias inativo) vs. Cloudflare R2 (só storage, 10GB grátis, zero egress) — não são concorrentes, são complementares.
- Pesquisado Cloudflare Pages vs. Vercel para Next.js 16: Cloudflare descontinuou "Pages" puro para Next.js; caminho atual é Workers+OpenNext, que não garante suporte completo no Windows (ambiente de dev do usuário). Recomendada Vercel.
- Apresentado o fork central da decisão via pergunta direta ao usuário: manter Fastify ou ir 100% Supabase; precisa de offline ou não. Usuário delegou a recomendação em ambos os casos.
- **Decidido (DEC-011, DEC-012, DEC-013):** modelo online-first single-user; Postgres no Supabase como única fonte de verdade; Fastify mantido como camada de API (não substituído pela API automática do Supabase); Supabase Auth para login; Supabase Storage para imagens (R2 como alternativa futura); Vercel para o frontend. DEC-003 e DEC-004 (Local-First + CRDT) formalmente superadas.
- **Código ajustado:** `packages/crdt` removido do monorepo; coluna `crdt_state` removida do `schema.ts`; nova migration (`0001_known_shinko_yamashiro.sql`) gerada e **validada contra um Postgres real** (instalado de novo no sandbox, que é efêmero entre sessões). Revalidados `pnpm typecheck` e `pnpm lint` nos 4 workspaces restantes — ambos passam limpo.
- Atualizada toda a documentação para refletir o pivô: `CONTEXT.md` (stack, armadilhas, contexto de produto), `ROADMAP.md` (F0 com itens de Supabase, F5 redefinida de "CRDT" para "Deploy em Produção"), `GLOSSARY.md` (termos obsoletos marcados como históricos), `STATUS.md`, `CHANGELOG.md`, `IDEAS.md` (ideia de compactação CRDT movida para Descartadas; nova ideia de fila offline simples registrada como adiada).
